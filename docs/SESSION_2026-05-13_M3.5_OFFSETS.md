Sessão de continuação do M3 (interactive overlay). Foco: portar o `Container` do SDK Linux pra ARM64 — descobrir offsets corretos, automatizar tradução de todos os structs, e parar de usar heap-scan heurístico.

- **Script `offsets_linux_to_arm64.py`**: parseia o `Offsets.h` Linux + 11 mappings verificados → gera `Offsets_android.h` com 532 offsets (11 VERIFIED, 3 CALIBRATED por calibração de gaps, 518 PREDICTED por simulação de layout)

## Como descobrimos os offsets do Container

1. **Estado in-game**: personagem "Tailegork" com HP 751/751, MP 100/100 visível na HUD
2. **`probe_reader` extendido com `search <decimal>`** procura u32 exato na heap inteira via `process_vm_readv`
3. **`adb shell su -c '/data/local/tmp/probe_reader 6160 search 751'`** retornou 64 hits, dos quais o entry **#23** bateu com:
   ```
   23  HP=751  HPMax=751  MP=100  MPMax=100  @0x76384dab35a4
   24  HP=751  HPMax=100  MP=100  MPMax=0    @0x76384dab35a8  [vt] vt=0x50103a0  cnt_start=HP-424
   ```
4. #23 e #24 são o mesmo Container, leitura deslocada em 4 bytes — #24 ficou 8-aligned e o scan de vtable funcionou
5. Vtable @ 0x050103a0 (em libwarspear .rodata) → **RVA = `0x050103a0 - 0x4380000 = 0xC903A0`**
6. Container começa em `0x76384dab3400` = `hp_addr - 0x1A4`. Logo **`Container::HP = 0x1A4`**
7. Dump do Container completo (`probe_reader 6160 76384dab3400 80`, etc.) revelou:
   ```
   +0x000: a0 03 01 05 00 00 00 00         ← vtable = 0x050103a0 ✓
   +0x080: pointer → +0x090                ← std::wstring data ptr
   +0x088: 13 00 00 00 09 00 00 00         ← cap=19, size=9
   +0x090: T a i l e g o r k               ← "Tailegork" UTF-32LE ✓
   +0x1A0: 07 31 00 00                     ← Allegiance (low byte=7)
   +0x1A4: ef 02 00 00                     ← HP=751 ✓
   +0x1A8: ef 02 00 00                     ← HPMax=751 ✓
   +0x1AC: 64 00 00 00                     ← MP=100 ✓
   +0x1B0: 64 00 00 00                     ← MPMax=100 ✓
   ```

## Tradutor automático Linux→ARM64

`helpers/offsets_linux_to_arm64.py`:

1. Parseia `ws-hack2/src/core/Offsets.h` extraindo (namespace, field_name, offset, comment) — 38 namespaces, 532 fields
2. Infere tipo de cada campo a partir do comentário (regex: "ptr", "u32", "wstring", "TreeNode", etc.)
3. Para cada namespace, ordena fields por offset e simula o layout ARM64:
   - ptrs: 4→8 bytes
   - wstring: 12→16 bytes
   - vector: 12→24 bytes
   - alinhamento 8 quando ptr/wstring
   - gaps entre fields tratados como `u32` (mesmo tamanho)
4. Para os 11 mappings VERIFIED de Container, **calibra** os gaps anteriores — se a predição diverge do verified, ajusta o cursor (assume que havia ponteiros escondidos no gap)
5. Emite `android-injector/src/game/Offsets_android.h` + `data/offsets_android.json`

**Resultado validado**: todas as 11 verified mappings batem exatamente com a predição após calibração.

```

       (apenas Container tem mappings VERIFIED hoje)
```

## PlayerWatch reescrito

Antes (M3): heap scan heurístico procurando tupla `(HP, HPMax, MP, MPMax)` com scoring + filtro de display dimensions + verificação opcional de vtable nearby.

Depois (M3.5): **scan por vtable**:

```cpp
uintptr_t base = findModuleBase("libwarspear.so");
uintptr_t vt_addr = base + 0xC903A0;  // Player vtable RVA
auto candidates = findByVtable(vt_addr);  // every heap obj with u64==vt_addr
```

Cada candidato é um Container válido (vtable bate). Leitura de stats:

```cpp
namespace C = OffsetsAndroid::Container;
uint32_t stats[4];
safeRead(container + C::HP, stats, 16);   // HP/HPMax/MP/MPMax adjacentes
auto name = readWideString(container + C::NameData,
                            safeU32(container + C::WStringLen));
```

Sem mais heurísticas, sem mais filtros de "display dimension". Determinístico.

## probe_reader extendido

```bash
adb shell "su -c '/data/local/tmp/probe_reader <PID> search <decimal_value>'"
```

Procura todo u32 alinhado a 4 bytes na heap (`/proc/PID/mem` via `pread64`) que iguala o valor passado. Pra cada match, dump (HP, HPMax, MP, MPMax) candidates + verificação de vtable em libwarspear .rodata nas 0x600 bytes anteriores.

Ferramenta crítica pra RE rápido em runtime — basta saber um valor visível no jogo (HP, level, gold, etc.) pra localizar a struct correspondente.

## Não funcionou ainda

Quando o user testou em-jogo no final da sessão, o scan **não pegou o Container correto** (Name continuou "(unknown)" ou stats não bateram com o jogo). Possíveis causas pra investigar:

1. **Vtable diferente por classe de personagem?** Tailegork é Necromancer. Outras classes (Mage, Ranger, etc.) podem ter vtables diferentes — multiple inheritance ou polimorfismo. Linha de investigação: rodar `probe_reader search <HP_atual>` em-jogo de novo, ver qual vtable aparece. Comparar com 0xC903A0.

2. **PIE relocation**: libwarspear base mudou entre sessões (0x4380000 → 0x4018000). RVA 0xC903A0 é relativa à base, então deveria funcionar — confirmar que `findModuleBase` retorna a base ARM64 (não a tradução Houdini).

3. **Filter rejeitando matches válidos**: `findByVtable` tem filtro `hp[1] > 200000 → skip` e `all-zeros → skip`. Pode estar dropando o Container correto em estados onde HP=0 (logout, morte, transição).

4. **vtable_addr não está nos primeiros 8 bytes**: confirmamos em uma sessão que estava, mas pode variar com herança virtual. Adicionar log do que está lá quando scan não acha nada.
